Here I have examples of Activities, Fractions, Data Classes, and their corresponding layout files. The app in these examples is structured using the Model-View-Controller architecture. It utilizes AndroidX Jetpack libraries to handle various task including saving data across process death or rotation with ViewModelProviders, RecyclerView to create a recycler view UI, and the Room library to create a database. Included in the database package is the database class itself, a DAO interface, and a type converter class.

Example Programs

  • CriminalIntent is an app that allows its users to document bad or irresponsible office behavior, name a suspect, and upload the crime to Face Book or Twitter. This app comes from the book Android Programming The Big Nerd Ranch Guid that I am currently reading. Among the things you can find demonstrated within this app are UI Fragments and the Fragment Manager, databases and the Room API, background threads, explicit and implicit intents, and much more. MainActivity is used to host two fragments: CrimeFragment, and CrimeListFragment, and uses the fragment manager to switch between the two. Its layout file is used as a container for the two fragments.


  • CrimeListFragment holds a list of crimes and displays them in a RecyclerView. The user can click on a crime in the list which takes them to detail screen of the app where they can and add details about the crime, such as a photo, date, title for the crime, and suspect. This Fragment consist of a TextView to hold the title of the crime, another TextView to hold the description of the crime and an ImageView to hold an image of handcuffs to show if the crime has solved. The user can also add a new crime by clicking on an image in the app bar labeled add crime.


  • CrimeFragment is the details part of the app. Once the user clicks on an existing crime or creates a new crime, CrimeFragment is inflated and the user can edit or add details of the crime. CrimeFragment consist of an EditText for the title of the crime, a Button to take a picture, an ImageView to hold the photo, a Button to select a date which calls a DatePicker DialogFragment, a suspect button to choose a suspect from the user's contacts, and a CheckBox to indicate if the crime has been solved. This information is then stored in a Room database using liveData as a background thread, and the photo is stored in a file on the local disc.


  • CriminalIntent is structured as a Model-View-Controller app and the model classes CrimeListViewModel and CrimeDetailViewModel hold the application's data and logic.


  • A "crime" consist of a title, the date the user observed the crime, a suspect, possibly a photo of the crime scene, and whether or not the crime has been solved. All of this data is stored in a Room database with the exception of the photos which are saved to a file on the local disc and referenced via the id property of the database. The database is held in a separate package with the classes CrimeDatabase to hold the database itself, CrimeTypeConverters to handle and convert data types and data being saved to the database, and an interface, CrimeDao, to handel the functions to query and insert data to the database.


  • CriminalIntent uses the Room library and MySQLite to create a database that stores the date the crime was recorded, the name of a potential suspect, the title of the crime, if the crime is solved, and a UUID for the crime. The database consists of CrimeDatabase, a class to hold the database itself, CrimeDao, a DAO interface to handle the MySQLite functions, a CrimeTypeConverters class to convert the data being saved and retrieved from the database, and CrimeRepository, a class to encapsulate the logic for accessing data from the database. Room uses Crime.kt to structure the database using @Entity and @PrimaryKey. liveData is used to save and retrieve data on a background thread.


  • CriminalIntent uses an additional DialogFragment, DatePickerFragment to give the user a date Dialog box to choose a date for the crime when they click on the date button.
